Spec Driven Developmentを補助するツールをアーキテクチャと設計思想の観点から比較・分析調査する。

## 対象ツール
- https://github.com/jasonkneen/kiro
- https://github.com/gotalab/cc-sdd
- https://github.com/github/spec-kit
- https://github.com/Fission-AI/OpenSpec
- https://github.com/classmethod/tsumiki
- https://github.com/Pimzino/spec-workflow-mcp
- https://github.com/bmad-code-org/BMAD-METHOD


## ルール
- 調査、分析結果を省略しないこと。
- リポジトリをローカル(`./sdd_tools`)にcheckoutして分析時点の最新コードに対して調査する。
- 各ツールについて以下を確認する：
  - READMEとドキュメント全体の精読
  - 主要なソースコード（特にコアロジック）の理解(全体のアーキテクチャ理解レベル)
  - Issue、PRから開発の方向性や課題を把握(web検索)
  - 公式ドキュメントがあれば参照(web検索)
- 必要に応じて外部情報の検索を行う。（公式ドキュメント、解説記事など）
- 情報が見つからない比較観点については、無理に推測せず『情報不明』または『該当する公開情報なし』と明記すること。


## 実行戦略（コンテキスト枯渇対策）

### フェーズ分割実行
各ツールを個別に分析し、中間成果物として保存することでコンテキストを効率管理する。

1. **フェーズ1**: 各ツール個別分析（1ツールずつ実行）
  -  `analysis/{tool-name}.md` に出力
    - 例：
      - kiro → `analysis/kiro.md`
      - cc-sdd → `analysis/cc-sdd.md`
      - spec-kit → `analysis/spec-kit.md`

2. **フェーズ2**: 統合レポート作成
   - 全ツールの個別分析を統合
   - 出力: `analysis-report.md`

### 各フェーズ実行ルール
- 1フェーズ完了後、次フェーズ実行前にユーザー確認を取る
- 個別分析は Task tool (subagent_type=Explore, thoroughness="very thorough") の活用を推奨
- 中間ファイルは必ず保存し、データ損失を防ぐ

### 分析深度レベル
各ツールは以下のレベルで分析を実施：

- **Level 1 (基礎調査)**: README + package.json/設定ファイル + ディレクトリ構造
- **Level 2 (標準調査)**: Level 1 + コアロジック解析 + 主要ドキュメント
- **Level 3 (詳細調査)**: Level 2 + 全ドキュメント + Issue/PR分析 + 外部情報検索

基本は Level 2 で実施し、必要に応じて Level 3 へ深堀りする。

## 出力フォーマット

### 個別分析ファイル（フェーズ1）
- GFM形式で `analysis/{tool-name}.md` に出力
- 「事実（分析結果）」と「解釈・意見（考察）」を明確に分けて記述
- フォーマット：
  - Front Matter
    - ツール名
    - リポジトリURL
    - 分析実施日(YYYY-MM-DD)
    - 分析深度レベル (Level 1/2/3)
    - 分析に利用したAIモデル
    - コミットハッシュ/日時
  - ツール概要（1-2段落）
  - 各比較観点ごとの分析結果
    - 調査・分析結果（事実ベース）
    - 考察（解釈・意見）
    - データソース（ファイルパス、URL、コミットハッシュなど）

### 統合レポート（フェーズ2）
- GFM形式で `analysis-report.md` に出力
- フォーマット：
  - Front Matter
    - タイトル
    - 分析実施日(YYYY-MM-DD)
    - 対象ツール（ツール全てのリポジトリURLのリスト）
    - 分析に利用したAIエージェント
    - 分析に利用したAIモデル
  - エグゼクティブ・サマリ（全体の要約を1-2段落で）
  - 各比較観点ごとの分析結果
    - サマリ（観点の要点を2-3文で）
    - 調査・分析結果（基本的に表形式、全ツール横断比較）
    - 結果の解釈・考察
    - データソース（個別分析ファイルへの参照）
  - 総合比較表（全ツールの主要特徴を一覧化）
  - 結論

## 比較観点

### 【必須】全ツール必須調査項目
以下は全ツールで必ず詳細調査すること：
- 基本コンセプト（ツールとしての思想）
- 核心技術
- ワークフロー
- 技術スタック
- プロジェクト構造
- 提供機能、ツール
  - コア機能（基本的な仕様駆動開発機能）
  - 差別化機能（独自の特徴的な機能）

### 【重要】可能な限り調査
情報が入手可能であれば詳細調査すること：
- 統制メカニズム(仕様（Spec）の変更、レビュー、承認に関するワークフローや仕組み)
- プロジェクト記憶(仕様の変更履歴や、その背景にある議論・決定をどのように記録・管理・参照するか)
- エコシステム
- ポータビリティ
- 導入アプローチ
- 既存プロジェクトに対する分析能力
- 言語対応

### 【補足】情報があれば追加
追加情報として調査（情報不足の場合は明記して省略可）：
- SWOT分析
- 実践的な選択基準
- ユースケース
- 課題、制約、注意点
